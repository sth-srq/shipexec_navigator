1. Purpose of the Application

ShipExec Navigator is a Blazor Server web application that serves as an AI-powered configuration management tool for ShipExec — a multi-carrier shipping software platform by PSI (Progistics). ShipExec manages shipping operations across enterprises, and its configuration is expressed as deeply nested XML documents containing companies, shippers, profiles, sites, users, business rules, and dozens of other entity types.

Navigator solves the following core problems:

  • Configuration Visualization: Renders the company XML as an interactive tree view (the "Navigator" panel) allowing operators to browse, search, and understand complex shipping configurations without manual XML parsing.
  • AI-Assisted Configuration Management: An integrated AI chat panel ("ShipExec Copilot") allows operators to issue natural-language commands — e.g., "delete all shippers in California" or "set all profile timeouts to 30 seconds" — which the AI translates into structured modification actions.
  • Diff & Apply Pipeline: All modifications produce a variance (diff) that shows exactly what will change before the operator commits. Changes are then applied to the live ShipExec Management Studio REST API with full audit logging.
  • Audit Trail: Every change is persisted to a SQL Server database with before/after snapshots, batch grouping, timestamps, and operator comments.
  • Template Management: Download, cache, and manage company configuration templates from the Management Studio.
  • Client-Specific Logic: A pluggable architecture allows per-customer behaviour overrides (e.g., WesBanco-specific field defaults or validation rules).
  • RAG-Enhanced Documentation Search: Operators can ask the AI questions about ShipExec concepts and features, which are answered using Retrieval-Augmented Generation backed by embedded ShipExec documentation.

3. How AI Is Used Throughout the Application

AI is deeply integrated into ShipExec Navigator at multiple levels. The primary AI integration is through Microsoft Semantic Kernel connected to Azure OpenAI.

3.1 AI Chat Panel ("ShipExec Copilot")

The main AI interface is the AiChatPage.razor page (route: / and /ask). This provides a conversational interface where operators interact with the AI to:

  • Ask questions about ShipExec configuration concepts (answered via RAG)
  • Search and filter entities (shippers, users, profiles, etc.)
  • Issue modification commands (add, edit, delete entities)
  • Analyse Client Business Rules (CBR scripts)
  • Search application and security logs
  • Generate JavaScript to manipulate the Navigator DOM (advanced use)

3.2 Semantic Kernel Plugin Architecture

The AI does not operate on raw text alone. Semantic Kernel's function calling mechanism allows the AI model to invoke typed plugins at runtime. The kernel is constructed per request (not singleton) with FunctionChoiceBehavior.Auto() so the model decides when to call plugins.

The Blazor frontend parses this JSON, renders the message as Markdown, and if an action is present, executes it — applying edits to the in-memory XML tree, showing confirmation dialogs, or running JavaScript.

Index Discovery at Runtime

InMemoryRagService looks for the index file in this priority order:

  1. {AppContext.BaseDirectory}/rag_index.json — primary (CopyToOutputDirectory)
  2. Configured path relative to the binary directory
  3. Configured path relative to working directory
  4. Configured path as absolute

If no index is found, the service logs a warning and returns empty results — the AI degrades gracefully to "no documentation found."

4.a How to Currently Add More Documents

Process to add new RAG documents:
  1. Place documents in the RAGDocuments/ folder (relative to the solution root). Supported formats: .txt and .pdf.
  2. Run the RAGLoader console app: 
    cd ShipExecNavigator.RAGLoader
dotnet run
  3. The loader will:
    • Discover all .txt and .pdf files recursively
    • Extract text (PdfPig for PDFs, direct read for text files)
    • Split into overlapping word-level chunks (500 words, 50 overlap)
    • Generate 384-dim embeddings locally (no API call)
    • Write the updated rag_index.json
  4. Copy the updated rag_index.json to the Blazor project's output directory (or rely on the CopyToOutputDirectory build action).
  5. Restart the Blazor app — InMemoryRagService loads the index once at startup.
Important: The RAGLoader and InMemoryRagService use the same local hash-projection embedding algorithm. If you change the embedding algorithm, you MUST re-run the RAGLoader to regenerate the index.

4.c RAG Deep Dive — Internals & Call Chain

This section documents the complete RAG pipeline from raw document to AI-augmented answer, including algorithm internals, data formats, and runtime orchestration.

4.c.1 The RagChunk Data Model

Every unit of indexed knowledge is represented as a single immutable record:

// ShipExecNavigator.RAGLoader\RagChunk.cs
internal sealed record RagChunk(
    string  Text,        // The chunk's raw text (up to ~500 words)
    string  Source,      // Relative file path within RAGDocuments/
    int     ChunkIndex,  // 0-based position within the source file
    float[] Embedding);  // 384-dimensional float vector

The serialized rag_index.json is simply a JSON array of these records:

[
  {
    "Text": "ShipExec supports ...",
    "Source": "Guide.pdf",
    "ChunkIndex": 0,
    "Embedding": [0.0234, -0.0812, 0.0491, ...]   // 384 floats
  },
  ...
]

4.c.2 The Chunking Algorithm

Text is split into word-level chunks with a sliding window approach:

static List<string> ChunkText(string text, int chunkSize, int overlap)
{
    var words = text.Split(' ', RemoveEmptyEntries | TrimEntries);
    int step  = Math.Max(1, chunkSize - overlap);  // stride = 450 by default

    for (int i = 0; i < words.Length; i += step)
    {
        var chunk = string.Join(' ', words.Skip(i).Take(chunkSize));
        chunks.Add(chunk);
        if (i + chunkSize >= words.Length) break;
    }
}
ParameterDefaultEffect
chunkSize500 wordsMaximum words included in a single chunk
chunkOverlap50 wordsContext overlap — prevents splitting a concept across two chunks
Stride (step)450 wordschunkSize − overlap; how far the window advances each iteration
Why overlap? If a concept spans the boundary between two consecutive windows, the 50-word overlap ensures at least one chunk contains the full thought, improving recall for queries that target boundary content.

4.c.3 The Local Embedding Algorithm (Hash-Projection)

Both the offline loader (LocalTextEmbeddingService) and the runtime query path (LocalEmbeddingGenerator) use an identical deterministic hash-projection algorithm. This ensures query embeddings are directly comparable to stored embeddings with no external API dependency.

// Simplified pseudocode of the 384-dim embedding generation
1. Tokenize text → lowercase, split on non-alphanumeric, discard tokens < 2 chars
2. Compute term-frequency: TF[token] = count of occurrences
3. For each unique token:
     weight = log(1 + TF[token])
     h1 = HashCode.Combine(token, 0) % 384   → target dimension 1
     h2 = HashCode.Combine(token, 1) % 384   → target dimension 2
     s1 = ±1 (random sign from hash)
     s2 = ±1 (random sign from hash)
     vector[h1] += s1 × weight
     vector[h2] += s2 × weight × 0.5
4. L2-normalize the 384-dim vector to unit length
PropertyValue
Dimensions384 (fixed)
API calls requiredZero — purely local computation
DeterministicYes — same text always yields same vector
Quality levelBag-of-words with term-frequency weighting (not contextual)
SpeedSub-millisecond per chunk/query
Tokenizer ruleSplit on non-alphanumeric; lowercase; min 2 chars
Critical invariant: LocalTextEmbeddingService (loader) and LocalEmbeddingGenerator (runtime) MUST implement the same algorithm. If either is changed, the entire index must be regenerated.

4.c.9 TopK Behavior & Performance

The plugin currently requests topK: 1000 from the search service, meaning it returns up to 1,000 matching chunks. This is an intentionally generous setting for the in-memory service where performance cost is negligible:

MetricValue (typical corpus of 500–2000 chunks)
Embedding generation (query)< 1 ms
Cosine similarity scan (full corpus)< 5 ms
Memory footprint per chunk~1.5 KB (384 floats × 4 bytes + text)
Total index memory (1000 chunks)~1.5 MB
Startup load time< 100 ms
Note on topK: 1000: While 1000 chunks may seem excessive, the LLM's context window handles the filtering — irrelevant low-scoring chunks at the tail are naturally ignored by the model. If token costs become a concern, reducing topK to 5–20 at the plugin level is a one-line change.

7. Example Chat Flow

The following illustrates the complete lifecycle of a single chat message from user input to UI update.

Step 1 — User types message
The operator types "Show me all shippers in Texas" into the <textarea> on AiChatPage.razor and clicks Send (or presses Enter).
Step 2 — Blazor component captures input
SendAsync() is invoked. The message is added to _messages list (role: "user"). A "thinking" animation is shown. The _input field is cleared.
Step 3 — Context assembly
The component gathers all available context:
  • xmlContext — the current company XML (if loaded via Navigator)
  • entityIndex — pre-built CompanyEntityIndex with all entity data
  • usersContext — serialized user JSON (if users panel is loaded)
  • cbrsContext — serialized CBR list (if CBRs are loaded)
  • logsContext — serialized log entries (if logs are loaded)
  • history — all previous messages in the conversation
Step 4 — Service call
IAiChatService.SendMessageAsync(history, userMessage, xmlContext, useRag, ...) is invoked.
The default implementation is SemanticKernelChatService.
Step 5 — Kernel construction
A fresh Kernel is built with Azure OpenAI chat completion. Plugins are registered based on available context (e.g., CompanyIndexPlugin if entity index exists, RagSearchPlugin if RAG is enabled). FunctionChoiceBehavior.Auto() is set.
Step 6 — System prompt assembly
A comprehensive system prompt is constructed with:
  • Persona instructions ("You are ShipExec Copilot…")
  • JSON response format specification
  • Available action types and payload schemas
  • Context-specific instructions (XML manipulation, user management, etc.)
  • Entity manifest (if index is available)
  • DOM cheat sheet for JavaScript generation
Step 7 — Chat history + user message sent to Azure OpenAI
chatService.GetChatMessageContentAsync(chatHistory, executionSettings, kernel, ct)
The model may invoke plugin functions (e.g., CompanyIndex/find_shippers) during reasoning. SK handles the tool-call loop automatically.
Step 8 — Response parsing
The response string is deserialised as AiChatResponse (JSON with message + optional action). If JSON parsing fails, it's treated as plain text.
Step 9 — UI update
The response message is added to _messages (role: "assistant"). If an action is present, the Blazor component processes it:

JS interop scrolls the chat container to show the newest message.

8.b What Happens on the Backend

  1. AI receives the command along with the entity index (or XML context)
  2. Semantic Kernel invokes plugin functions — e.g., CompanyIndexPlugin.find_shippers or CompanyIndexPlugin.edit_shippers to identify matching entities and determine edits
  3. AI formulates the structured action — returns JSON with action type and payload containing entity IDs, names, and edit details
  4. Blazor component receives the response — parses the AiChatResponse
  5. In-memory XML tree is modified — the component locates the matching XmlNodeViewModel nodes in the tree and applies the field changes directly to the in-memory model
  6. Variance detection (on explicit "Apply") — when the operator clicks "Apply Changes":
    • The modified XML is serialized back to a string
    • ShipExecService.GetDiffAsync(originalXml, modifiedXml) is called
    • This calls AppManager.GetVariancesAndRequests() which deserializes both XMLs into Company object graphs
    • CompanyBuilderManager.GetVariances() compares them field-by-field across all entity types
    • Each variance generates a typed RequestBaseWithURL (the HTTP request to push the change)
  7. Apply to live API — CompanyBuilderManager.ApplyRequests() executes each request against the Management Studio REST API with the operator's Bearer JWT
  8. Audit logging — Each applied variance is persisted to dbo.Variances via VarianceManager.InsertAsync() with before/after JSON snapshots, batch ID, and timestamps

8.c How the Front End Gets Updated

  1. Immediate visual feedback: When the AI returns an action, the matching tree nodes are updated in the XmlNodeViewModel tree immediately. The Blazor component re-renders via StateHasChanged().
  2. Change indicators: Modified nodes are visually marked (different styling/badge) to show pending changes not yet applied to the live system.
  3. Diff dialog: Before applying, the operator can view a detailed diff (DiffModal.razor / DiffResultDialog.razor) showing field-by-field changes with old → new values.
  4. Apply results: After applying, per-entity success/failure results are shown. The original XML is refreshed to reflect the new state.
  5. Chat message: The AI's message field (rendered as Markdown) explains what was found/changed in human-readable form.

Key Design Decisions

DecisionRationale
Kernel built per-request (not singleton)Each user message gets a fresh plugin context. Prevents state leakage between requests. Plugins receive current XML/index snapshot.
Scoped service lifetime for ShipExecServiceEach SignalR circuit (browser tab) gets independent state — connected company, cached XML, pending variances. No cross-tab interference.
Local hash-projection embeddings (no API calls for RAG)Zero latency for vector search. No API cost. Deterministic. Acceptable quality for keyword-proximity search on a focused documentation corpus.
Structured JSON response format from AIEnables deterministic action parsing. The AI returns both a human-readable message AND a machine-parseable action in one response.
Lazy tree loading (skeleton + on-demand categories)Company XMLs can have thousands of entities. Loading everything upfront would be slow. Categories are fetched only when expanded.
Variance pipeline (diff before apply)Safety mechanism — operator always sees exactly what will change before committing. Full audit trail with before/after snapshots.
Client-specific logic via strategy patternClean separation of per-customer customizations. Adding a new client = one new class + one if branch in ClientLogicResolver.
IVectorSearchService abstractionFuture-proofs for Qdrant/Pinecone/etc. migration. One-line DI swap.

Security Considerations

  • JWT-based auth: All Management Studio API calls use Bearer tokens. No credentials are stored server-side.
  • Content Security Policy: Strict CSP headers applied (script-src, style-src, frame-ancestors).
  • No inline secrets: API keys in appsettings.json (environment-overridable). Azure Key Vault integration possible.
  • Audit trail: Every configuration change is logged with full before/after state in SQL Server.
  • HTTPS enforced: HSTS + HTTPS redirection in production.

Scalability Path

  • RAG: Local → Qdrant → Azure AI Search (progressive, interface-driven)
  • Embeddings: Hash projection → text-embedding-ada-002 → custom fine-tuned model

  • Database: Local SQL Server → Azure SQL → Cosmos DB (if needed)
  • Hosting: Single App Service → multiple instances behind Azure Front Door

Matching Algorithm

The IsNonEditable(nodePath) method checks whether the node path (or any prefix of it) matches a configured path using case-insensitive comparison. Because matching checks every prefix of length ≥ configured depth, any descendant of a non-editable node is also non-editable.

I.3 Phantom Update Suppression

XML serialization of complex properties (arrays, nested objects) can produce cosmetic differences that don't represent real user edits. These "phantom updates" are detected and hidden from the display:

// A phantom is an Update variance with no displayable scalar-field changes
private static bool IsPhantomUpdate(VarianceInfo vi) =>
    vi.ChangeType == "Update"
    && !string.IsNullOrEmpty(vi.OriginalXML)
    && !string.IsNullOrEmpty(vi.NewXML)
    && FieldDiffHelper.GetChangedFields(vi.OriginalXML, vi.NewXML).Count == 0;
ScenarioResult
User changes Address1 on a shipperShown as Update with field diff: Address1 "old" → "new"
Array property serializes differently but no scalar changePhantom — suppressed from display
Phantom variance existsHidden from UI but API request is still generated correctly

I.4 Child Variances (Site Aggregation)

Sites contain nested sub-entities (Machines, PrinterConfigurations, ScaleConfigurations). When these change, the variance is grouped under a parent Site variance:

Site: Chicago [Update]
├── Machine: Printer-01 [Update] — field changes shown
├── ScaleConfiguration: Scale-A [Remove]
└── PrinterDefinition: Label-4x6 [Add]

The parent variance drives the API request (e.g. UpdateSite), while child variances are display-only for user clarity.

I.5 Selective Apply

Users can select which variances to apply via checkboxes. Only selected indices are passed to ApplyChangesAsync(selectedIndices, comments). This allows partial application — e.g. apply shipper changes now, defer user changes for later.

Appendix J — SignalR Circuit Management & Reconnection

As a Blazor Server application, all UI interactions are mediated through a persistent SignalR WebSocket connection (a "circuit"). If the connection drops, the user must reconnect or the session is lost.

J.1 Service Lifetime Model

LifetimeServicesImplication
Scoped (per-circuit)ShipExecService, IShipExecServiceEach browser tab gets its own AppManager instance with its own JWT and company context. No cross-tab state leakage.
SingletonInMemoryRagService, IVectorSearchServiceRAG index loaded once at app startup and shared across all circuits.
TransientSK plugins, KernelBuilt per-request to avoid stale state between AI calls.

J.2 Reconnection Modal

The ReconnectModal.razor component handles all connection-loss scenarios:

StateUser ExperienceAction
showModal appears: "Rejoining the server..."Automatic reconnect attempt begins
hideModal closesCircuit restored, UI continues normally
failed"Rejoin failed... trying again in X seconds"Countdown + auto-retry when tab becomes visible
rejectedN/AFull page reload (circuit ID unknown to server)
pause"The session has been paused by the server""Resume" button available

J.3 Reconnection Logic

// Simplified reconnection flow:
1. Blazor.reconnect() → attempts WebSocket reconnection
   - true:  success, modal closes
   - false: server reachable but circuit unknown
     → Blazor.resumeCircuit() → attempts circuit resume
       - true:  resumed, modal closes
       - false: page reload (fresh start)
   - exception: server unreachable
     → wait for tab visibility change, then retry
Session data on disconnect: Because ShipExecService is scoped to the circuit, a failed reconnection means all in-memory state (loaded XML, entity index, pending variances) is lost. The user must re-connect to ShipExec and re-select their company. This is by design — the server doesn't persist mid-session state.
